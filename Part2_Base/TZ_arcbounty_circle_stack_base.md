# ТЗ: ArcBounty × Circle Agent Stack + Base mainnet

**Версия:** 1.0 (13.07.2026)
**Исполнитель:** Claude Code
**Репозитории:** контракты ArcBounty (Arc testnet), frontend (arcbounty.app), arcbounty-agent-sdk

## Контекст

ArcBounty — децентрализованный bounty-маркетплейс на Arc Network (ERC-8183 / ERC-8004), эскроу в USDC на смарт-контрактах, рабочий фронт и SDK, реальные сделки на Arc testnet. Цель этого ТЗ — четыре блока:

1. Платный фасадный REST API (x402) — вход в Circle Agent Marketplace.
2. Поддержка Circle Agent Wallets в arcbounty-agent-sdk.
3. SKILL.md для агентов (формат Agent Skills).
4. Деплой контрактов на Base mainnet (мультичейн: Arc testnet + Base).

Блоки независимы, порядок выполнения: 4 → 1 → 3 → 2 (Base-деплой первым, чтобы фасад и e2e-тесты могли работать против mainnet-бэкенда, а не только testnet).

**Важно:** протокол и контракты НЕ меняем. x402 — это только оплата вызовов фасадного API, он не трогает эскроу-логику.

## Блок 1. Фасадный API с x402 (оценка: 1–2 дня)

### Назначение
Публичный платный REST поверх ArcBounty, чтобы (а) пройти листинг в Circle Agent Marketplace — курируемый каталог сервисов, оплачиваемых агентами в USDC; (б) агенты с кошельками могли программно находить и постить баунти.

### Стек
TypeScript, Fastify (или Express — согласуй с тем, что уже в кодовой базе), Docker. Отдельный пакет/папка `apps/facade-api` в монорепо или отдельный репо `arcbounty-api` — выбери по текущей структуре.

### Эндпоинты v1

| Метод | Путь | Что делает | Цена x402 |
|---|---|---|---|
| GET | /health | статус, версия, поддерживаемые сети | бесплатно |
| GET | /openapi.json | OpenAPI 3.1 спека | бесплатно |
| GET | /v1/bounties | поиск: status, tags, minReward, maxReward, chain, пагинация | $0.001 |
| GET | /v1/bounties/:id | детали + статус эскроу + дедлайны | $0.001 |
| GET | /v1/bounties/:id/submissions | список сабмишенов (публичные поля) | $0.001 |
| POST | /v1/bounties/prepare | подготовка транзакции создания баунти | $0.01 |

Discovery-эндпоинты (/health, /openapi.json) обязаны быть бесплатными — агент должен уметь понять сервис до оплаты.

### Паттерн POST /v1/bounties/prepare
Фасад НЕ кастодиальный и НЕ релеер. Эндпоинт принимает параметры баунти (title, description hash/URI, reward, deadline, chain), валидирует и возвращает готовую неподписанную транзакцию: `{ to, data, value, chainId }` + человекочитаемое описание. Агент подписывает своим кошельком сам. Плата x402 — за сервис подготовки/валидации, не за эскроу.

### x402-обвязка
```ts
import { createGatewayMiddleware } from "@circle-fin/x402-batching/server";

const gateway = createGatewayMiddleware({
  sellerAddress: process.env.SELLER_ADDRESS!, // кошелёк Софии
});

app.get("/v1/bounties", gateway.require("$0.001"), listBountiesHandler);
```
Поведение: запрос без оплаты → HTTP 402 с платёжными инструкциями по спецификации x402; с оплатой → 200. Точную сигнатуру сверить в Блоке 0.

### Данные
Источник — чтение из контрактов (RPC) + существующий индексер, если он есть в репо. Кэш ответов 15–30 сек (in-memory или Redis, Redis уже в стеке проектов). Логирование: request id ↔ tx hash оплаты — пригодится для отчётности по грантам.

### Деплой
Dockerfile + docker-compose. Целевой сервер — существующий Ubuntu-сервер проекта (порт согласовать, за reverse proxy с TLS). `.env.example` со всеми переменными.

### Acceptance criteria
- [ ] `curl` без оплаты на платный роут → 402 с корректным x402-телом.
- [ ] Оплата тестовым кошельком через Circle CLI → 200, валидный JSON.
- [ ] /openapi.json проходит валидацию OpenAPI 3.1.
- [ ] README фасада: как агенту оплатить вызов (сниппет с Circle CLI).
- [ ] p95 латентность GET-роутов < 800ms при тёплом кэше.

## Блок 2. Circle Agent Wallets в arcbounty-agent-sdk (оценка: 2–4 дня, самый большой)

### Назначение
Агент с Circle Agent Wallet (permissionless, policy-controlled кошельки для агентов) может брать баунти, сдавать работу и получать выплату через наш SDK. В грантовой деке это milestone «Circle Wallets connector» — в коде и доках переименовать в **Agent Wallets support**.

### Задачи
1. Изучить интерфейс подписанта, который SDK использует сейчас (viem account / ethers signer — по факту кодовой базы).
2. Модуль `CircleAgentWalletSigner`, реализующий этот интерфейс поверх Circle Agent Wallet (через официальный SDK/CLI-механику Circle — по результатам Блока 0). Покрыть операции: `takeBounty`, `submitWork`, `claimPayout`.
3. Пример конфигурации policy-гардрейлов кошелька (лимит на транзакцию / в день) в `examples/`.
4. E2E-скрипт `examples/circle-agent-e2e.ts`: создать кошелёк → профинансировать → найти баунти через фасадный API (связка с Блоком 1) → взять → сдать → получить payout. Сеть: Arc testnet; если Agent Wallets не поддерживают Arc — Base (см. Блок 0), это зафиксировать.
5. Обновить README SDK + грантовую деку (переименование milestone).

## Блок 3. SKILL.md «arcbounty» (оценка: полдня)

### Назначение
Скилл в формате Agent Skills (открытый стандарт: SKILL.md с YAML-фронтматтером name/description + инструкции), чтобы любой кодинг-агент (Claude Code и др.) умел работать с ArcBounty. Дистрибуция: свой репо `arcbounty/skills`, установка `npx skills add <repo>`, публикация в каталоге skills.sh. В circlefin/skills PR не делаем — там только продукты Circle.

### Содержание скилла
- Что такое ArcBounty: роли (poster / worker-агент), жизненный цикл баунти, эскроу и dispute-флоу.
- Workflow для агента-исполнителя: discover (через фасадный API с x402 или напрямую из контрактов) → оценка задачи → take → submit → payout.
- Workflow для постера: prepare → sign → fund escrow.
- Типовые ошибки: USDC = 6 децималей; на Arc газ — USDC (не ETH); поля дедлайнов; повторный take; что происходит при просрочке.
- Ссылки: адреса контрактов по сетям (Arc testnet, Base), OpenAPI фасада, репо SDK.

### Acceptance criteria
- [ ] Скилл ставится через `npx skills add` и подхватывается Claude Code.
- [ ] Прогон: свежая сессия Claude Code со скиллом успешно выполняет сценарий «найди открытые баунти и покажи топ-3 по reward» через фасадный API.

### Задачи
1. Ревизия хардкодов сети в контрактах/скриптах деплоя (USDC-адрес, реестры 8004) → вынести в конфиг по chainId.
2. Деплой контрактов на Base mainnet (8453). Сначала полная репетиция на Base Sepolia (84532).

4. **Safety cap (обязательно):** пока нет аудита, ограничение размера баунти на уровне контракта — `maxBountyAmount`, дефолт 500 USDC, изменяемый owner'ом. На фронте — плашка «unaudited beta, cap $500». Это осознанное решение: mainnet без аудита допустим только с ограничением ущерба.
5. Верификация контрактов на Basescan.
6. Frontend: переключатель сетей (Arc testnet / Base mainnet), SDK: мультичейн-конфиг.
7. Индексер/subgraph (если есть) → добавить Base.
8. `docs/DEPLOYMENTS.md`: таблица адресов по сетям, дата, коммит.
